second program --> monitor-rw-nonstarve.c

In this program, the goal was to eliminate writer starvation. To achieve this, I introduced a writer_counter variable 
to track the number of writers waiting and determine if a writer should be given priority.
The solution ensures that readers can only start reading if there are no waiting writers or if a writer is already writing. 
This prevents new readers from continuously delaying writers.
Writers, on the other hand, are only allowed to write when there are no active readers and no other 
writer is currently writing. This guarantees that writers do not get indefinitely delayed by incoming readers.

third program --> monitor-gpu.c
In this program, the task was to create a system that allows only four threads to use the GPU simultaneously, 
ensuring that all four threads share the same dataset.
If a thread does not have the same dataset as the others, it must go to sleep and wait until it is allowed to
 run with a matching dataset.

fourth program --> monitor-dining.c
In this program, we were tasked with solving the Dining Philosophers' Dilemma. 
This one was pretty interesting, but Professor Thain gave us a great hint during class. 
To break the Dining Philosophers' Dilemma, we need to break the cycle that occurs if we start from the 
first philosopher and sequentially go to the last one. If we do this, it could lead to the last philosopher 
creating a cycle because they will all be waiting for the right fork, which will eventually cause a deadlock.

To reverse this, Professor Thain told us to break the cycle. In this case,
when we get to the first philosopher, we have them grab the right fork first and then the left fork second (themselves).
This will break the cycle (if we go sequentially) because the last philosopher will no 
longer have to wait for the right fork. So, it’s important to ensure that we can prevent deadlock if this happens.

In this output, we can see that we start with our philosophers sleeping, but philosopher 0 gets hungry 
and starts eating, which means it will block fork 2. Then, we let philosopher 4 eat because philosopher 0 is done 
eating and goes to sleep. After that, we let philosopher 4 eat because fork 0 (which creates a circular motion) 
is now available for philosopher 4. We can also let philosopher 2 eat because its forks are not blocked by philosopher 1, 
and so on. Once philosopher 2 is done and sleeping, we allow philosopher 3 to eat.

The implementation I did here was to assign every philosopher their left and right fork first. 
However, if the current philosopher is the last one, they are assigned the right fork first and then the left fork, 
which breaks any cycles and prevents deadlocks. We also had to track which forks were taken, which is how we update 
the fork data structure. Whenever we allow a philosopher to eat because their forks are available, we mark these forks
 as used by placing a -1 in the indices of those forks. Once the philosopher finishes eating, we access these forks again 
 and change them back to 0 to signal that the forks are free and available for use.
